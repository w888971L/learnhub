# Reference Note: Designs

*Early-stage thinking that hasn't graduated to a formal plan yet.*

**When to consult:** When brainstorming new capabilities, evaluating whether an idea has already been sketched, or looking for the project's overarching mission.

**Location:** `docs/designs/`

## Contents

- **`mission.md`** — the overarching vision: human operators and AI agent teams managing large codebases through constitutional governance
- **`prototypes/pilot-tests.md`** — pre-implementation testing to validate plan assumptions before coding
- **`prototypes/bites.md`** — mobile-friendly micro-briefings for quick operator orientation
- **`prototypes/mobile-operator.md`** — managing the full agent team from a phone via text conversation
- **`prototypes/protocol-vocabulary.md`** — domain-specific language for operator-agent communication modes
- **`prototypes/inspector-general.md`** — formidable independent enforcer with an operator-defined mandate the team doesn't see
- **`prototypes/outlier-detection.md`** — specialized enforcers that spot structural anomalies in the knowledge graph
- **`prototypes/zoom.md`** — visual zoom interface for navigating the knowledge graph from domains down to code
- **`prototypes/cloud-constitution.md`** — storing a synthetic constitution in Claude.ai project files, decoupling governance from the repo
- **`prototypes/librarian-analogy.md`** — three-role analogy (reader, librarian, auditor) for communicating orientation-over-retrieval

## Relationship to Plans

Designs are not plans. They have no template, no review process, no decision authority. A design becomes actionable only when it graduates to a formal plan in `docs/plans/`.
